# DEEPNOID PPT Generator — 설계 문서

- **작성일:** 2026-05-16
- **상태:** 확정 (구현 계획 수립 대기)
- **저장소:** `deepnoid-ppt-generator` (GitHub, Private)

---

## 1. 개요

Excel·Word 등의 문서를 입력하면, DEEPNOID 공식 PPT 템플릿 디자인을
**픽셀 단위로 그대로 유지한 채** 발표용 `.pptx`를 자동 생성하는 Claude Code 플러그인.

플러그인은 단순한 스킬이 아니라, 생성 과정을 지휘하는 **에이전트 팀**을 포함한다.
에이전트 팀은 입력 콘텐츠를 맥락에 맞는 템플릿 슬라이드에 적절히 배치하도록 조율한다.

## 2. 목표와 비목표

**목표**
- 입력 문서(Excel/Word)의 내용을 DEEPNOID 템플릿 슬라이드로 자동 구성한다.
- 색상·폰트·레이아웃·로고 위치 등 모든 디자인 요소를 원본과 동일하게 보존한다.
- 사용자가 생성 전에 슬라이드 구성(아웃라인)을 검토·수정할 수 있다.
- 플러그인은 GitHub 저장소로 배포하며, 비개발자도 README만 보고 사용할 수 있다.

**비목표 (이번 범위 아님)**
- 새 디자인 창작 — 템플릿을 벗어난 슬라이드는 만들지 않는다.
- 가이드 페이지(원본 1~18번)와 그리드 설명 페이지(23~26번)는 출력 대상이 아니다.
- 적용 예시 슬라이드(원본 32~46번)는 이번 카탈로그에 포함하지 않는다 (핵심 제네릭만).
- PowerPoint 슬라이드 마스터/레이아웃 재설계는 하지 않는다.

## 3. 입력 자료

- `DEEPNOID 2026 PPT 작성 템플릿.pptx` — 원본 57장 템플릿.
- `딥노이드 브랜드 디자인 가이드_260130_v9_RGB_Full.pdf` — 브랜드 디자인 가이드.
- 원본 57장은 유형별로 분류 완료 (`slides_by_type/`): Cover·Index·Divider·Guide·Layout·Content·Chart·Table·Closing.

## 4. 핵심 결정사항

| 항목 | 결정 | 이유 |
|---|---|---|
| 입력→슬라이드 매핑 | **아웃라인 먼저 검토** | 잘못된 매핑을 생성 전에 사용자가 잡을 수 있음 |
| 차트 처리 | **내장 차트 재바인딩** | 차트 슬라이드를 통째 복제 후 데이터만 교체 → 차트 스타일 100% 보존 |
| 런타임 | **Python 설치 후 Python 기반** | python-pptx·openpyxl·python-docx 등 검증된 오피스 처리 스택 |
| 템플릿 범위 | **핵심 제네릭 템플릿 10종만** | 깨끗한 범용 레이아웃 위주, 구성 단순 |
| 지휘 구조 | **지휘자 + 4 전문 에이전트** | 역할 분리 명확, 슬라이드 편집 병렬화 이득 |
| 생성 방식 | **템플릿 클론 파이프라인** | 디자인 픽셀 보존의 유일한 안전한 방법 (아래 5절) |

## 5. 생성 방식 — 템플릿 클론

원본 `.pptx`는 XML 파일들의 zip이다. 생성 방식:

1. 마스터 템플릿 `.pptx`를 복사한다.
2. 압축을 풀어 슬라이드 XML에 접근한다.
3. 카탈로그에 등록된 템플릿 슬라이드를 **통째로 복제**한다.
4. 복제본 안의 **텍스트·차트 데이터만** 교체한다.
5. 다시 압축해 `.pptx`로 만든다.

배경·로고·도형·색상·폰트·좌표는 원본 XML 그대로이므로 바이트 단위로 보존된다.

**기각한 대안:** 디자인 슬라이드를 PowerPoint `slideLayout` 마스터로 변환 후
python-pptx로 새로 그리는 방식. 변환이 손실적이고 디자인 드리프트 위험이 있어
"1픽셀도 변경 금지" 요구와 충돌 → 기각.

## 6. 충실도 보장과 한계

- **보장:** 정적 디자인 요소(배경·로고·도형·색·폰트·좌표)는 100% 동일.
- **유일한 변수:** 텍스트 길이. 입력 내용이 원본 텍스트 박스보다 길면 넘침/줄바꿈 발생.
  이는 디자인 변경이 아니라 콘텐츠 오버플로우다.
- **완화책 (2중):**
  1. 아웃라인 검토 단계 — 카탈로그의 슬롯 용량(`capacity`)을 초과하면 사전 경고.
  2. 검수 단계 — 렌더링 후 오버플로우를 잡아 자르기/슬라이드 분할로 해결.

## 7. 아키텍처 — 플러그인 구조

```
deepnoid-ppt-generator/            # 저장소 루트 = 플러그인 루트
  .claude-plugin/plugin.json       # 플러그인 매니페스트
  commands/
    generate-ppt.md                # /generate-ppt — 지휘자 진입점
  agents/
    ppt-conductor.md               # 지휘자
    content-analyst.md             # 콘텐츠 분석가
    layout-architect.md            # 레이아웃 설계자
    slide-assembler.md             # 슬라이드 조립가
    fidelity-qa.md                 # 충실도 검수자
  skills/generate-deepnoid-ppt/
    SKILL.md                       # 파이프라인 메커니즘 + XML 편집 규칙 (공유 지식)
    assets/
      deepnoid-template.pptx       # 마스터 템플릿 (10장 추출본)
      slide-catalog.json           # 템플릿 슬라이드 카탈로그
      catalog-previews/*.png       # 유형별 미리보기 이미지
    scripts/                       # Python 헬퍼 스크립트
  docs/
    specs/                         # 설계 문서
    reference/                     # 원본 57장 분류 이미지 (레퍼런스)
  README.md  LICENSE  .gitignore
```

**스킬과 에이전트의 분담:**
스킬(`SKILL.md`)은 *방법론* — 압축 해제·클론·XML 편집·차트 재바인딩·재압축 절차,
카탈로그, 스크립트 — 을 담은 공유 지식. 에이전트 팀은 그 지식을 *실행*한다.

## 8. 에이전트 팀 (5)

| 에이전트 | 역할 | 입력 → 출력 |
|---|---|---|
| **지휘자** `ppt-conductor` | 5단계 파이프라인 총괄, 사용자 대화(아웃라인 승인), 전문가 디스패치 | 입력 문서 → 최종 `.pptx` |
| **콘텐츠 분석가** `content-analyst` | Excel/Word 파싱 | 문서 → `content.json` |
| **레이아웃 설계자** `layout-architect` | 맥락에 맞는 템플릿 배치 — 콘텐츠↔템플릿 매핑, 아웃라인 작성, 슬롯 용량 경고 | `content.json` + 카탈로그 → `outline.json` |
| **슬라이드 조립가** `slide-assembler` | 카탈로그 슬라이드 클론 + 텍스트/차트 XML 교체 (슬라이드별 병렬 실행) | 승인된 아웃라인 → 편집된 슬라이드 XML |
| **충실도 검수자** `fidelity-qa` | 렌더링 후 오버플로우·충실도·잔여 플레이스홀더 점검 | 조립된 `.pptx` → 이슈 리포트 |

**지휘 메커니즘:** Claude Code에서 서브에이전트는 다른 서브에이전트를 디스패치하지
못한다. 따라서 "지휘자"는 `/generate-ppt` 명령으로 **메인 세션에서 실행**되며,
거기서 4명의 전문 서브에이전트를 순서대로(조립가는 슬라이드별 병렬로) 호출한다.
`ppt-conductor.md`는 그 지휘 로직을 정의한 문서다.

## 9. 슬라이드 카탈로그 (10종)

원본 57장 분류 결과에서 가장 깨끗한 슬라이드를 선정한다.

| # | 카탈로그 유형 | 원본 슬라이드 | 용도 |
|---|---|---|---|
| 1 | `cover` | 20 | 표지 — 메인 타이틀 |
| 2 | `index` | 21 | 목차 (Contents) |
| 3 | `divider` | 22 | 섹션 간지 — 큰 번호 + 타이틀 |
| 4 | `content-1col` | 27 | 1단 그리드 본문 |
| 5 | `content-2col` | 28 | 2단 그리드 — 수치 콜아웃형 |
| 6 | `content-3col` | 29 | 3단 그리드 본문 |
| 7 | `content-4col` | 30 | 4단 그리드 — 카드형 |
| 8 | `chart` | 38 | 내장 차트(막대) 슬라이드 |
| 9 | `table` | 46 | 표 슬라이드 |
| 10 | `closing` | 57 | 마무리 — 문의처 |

**마스터 템플릿:** 원본 57장에서 위 10장만 추린 `deepnoid-template.pptx`를
플러그인에 포함한다. 테마·슬라이드 마스터·레이아웃·임베디드 폰트 등 공유 리소스는
모두 보존하므로 디자인 충실도에 영향이 없다. 원본 57장 파일은 레퍼런스로 별도 보관.

## 10. `slide-catalog.json` 스키마

각 카탈로그 유형은 다음을 기록한다.

```json
{
  "type": "content-2col",
  "sourceSlide": 5,
  "description": "수치 지표 2~6개를 큰 숫자로 강조할 때",
  "preview": "catalog-previews/content-2col.png",
  "slots": [
    { "slotId": "title", "role": "title", "anchor": "<도형 식별자>",
      "capacity": { "maxChars": 40, "maxLines": 1 } },
    { "slotId": "stat-N", "role": "statValue", "anchor": "...",
      "repeatable": true, "min": 2, "max": 6 }
  ],
  "chartSpec": null,
  "tableSpec": null
}
```

- `sourceSlide` — 추출된 마스터 템플릿(`deepnoid-template.pptx`, 10장) 내의 슬라이드
  번호(1~10)다. 9절 표의 카탈로그 `#` 와 같다. (9절의 "원본 슬라이드" 열은 추출 전
  57장 기준 번호이며 추적용 참고값일 뿐, 카탈로그에는 추출본 번호를 쓴다.)
- `slots` — 편집 가능한 콘텐츠 자리. `capacity`로 설계자가 오버플로우를 사전 경고.
- `repeatable` — 콜아웃/카드처럼 개수가 가변인 슬롯. 원본보다 적으면 잉여 도형을
  통째로 삭제한다 (텍스트만 비우지 않음).
- `chartSpec` — 차트 슬라이드는 차트 종류·계열 수·카테고리 수를 기록 →
  재바인딩 시 데이터 형태를 검증한다.
- `tableSpec` — 표 슬라이드는 설계된 행×열 기준을 기록한다.

**카탈로그 작성:** 플러그인 제작 시 10장의 템플릿 XML을 직접 분석해 각 슬롯의
위치(anchor)·용량(capacity)을 손으로 기록하는 1회성 작업이다.

## 11. 파이프라인 데이터 흐름

| 단계 | 담당 | 입력 → 산출물 |
|---|---|---|
| 1. 분석 | 콘텐츠 분석가 | 입력 문서 → `content.json` (섹션·문단·표·수치 계열) |
| 2. 설계 | 레이아웃 설계자 | `content.json` + 카탈로그 → `outline.json` |
| — 검토 — | 지휘자 ↔ 사용자 | 아웃라인 제시 → 사용자 수정 → **승인된 아웃라인** |
| 3. 조립 | 슬라이드 조립가 (병렬) | 승인 아웃라인 → 편집된 슬라이드 XML |
| 4. 재압축 | 지휘자 | 정리 + 재압축 → 드래프트 `.pptx` |
| 5. 검수 | 충실도 검수자 | 드래프트 → 이슈 리포트 → (이슈 시) 3단계 재편집 루프 |

`outline.json` 항목 예: `{ 슬라이드 순번, 카탈로그 유형, 원본 슬라이드, 콘텐츠 매핑, 용량 경고[] }`

## 12. QA 메커니즘

- 검수 대상: 텍스트 오버플로우, 잔여 플레이스홀더, 차트/표 데이터 정합성.
  (정적 디자인은 클론이라 검수 불필요.)
- 렌더링: `soffice`(pptx→pdf) + Node 렌더러(pdf→png). 두 도구 모두 현재 환경에서
  동작 확인됨 — poppler 불필요.
- 이슈 발견 시 해당 슬라이드만 조립가에게 재디스패치 → 재압축 → 재검수.
  클린 패스가 나올 때까지 반복.

## 13. 의존성과 환경

- **Python** — 설치 필요. python-pptx, openpyxl, python-docx, markitdown.
- **Node.js** — 설치됨. PDF→이미지 렌더링 보조.
- **LibreOffice (`soffice`)** — 설치됨. pptx→pdf 변환.
- 플러그인 제작 시 의존성 설치 절차를 README와 SKILL.md에 명시한다.

## 16. 구현 시 결정할 항목

- 입력 문서에서 표/수치 데이터를 식별하는 규칙 (어떤 표를 `table` 슬라이드로,
  어떤 수치를 `chart`로 보낼지) — 레이아웃 설계자의 휴리스틱으로 정의.
- 차트 슬라이드(원본 38번)의 내장 차트 종류·계열 구조 — 카탈로그 작성 시 XML 확인.
- 표 슬라이드(원본 46번)의 행 추가/삭제 방식 — 카탈로그 작성 시 XML 확인.
- 입력 문서가 어떤 템플릿 유형에도 맞지 않을 때의 처리 — 설계자가 사용자에게 확인.
